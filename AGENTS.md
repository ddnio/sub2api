# AGENTS.md

本文件定义本仓库中人类开发者和编码代理都应遵守的工作规则。

## 目的

- 让仓库级规则保持简短、稳定、可执行。
- 把详细流程下沉到 `docs/engineering/`。
- 能交给自动化和 CI 执行的规则，尽量不要只停留在文档里。

## 规范分层

- 代理协作规则放在本文件。
- 详细 Git 工作流放在 `docs/engineering/git-workflow.md`。
- 上线、运营、执行计划放在 `docs/plans/` 和 `docs/ops/`。
- 不会影响日常工程协作的产品说明和部署细节，不要重复写在这里。

## 远端模型

- `origin` 必须指向团队自己的 fork。
- `upstream` 必须指向原始上游仓库。
- 不要直接基于 `upstream` 开发和推送。

## 分支模型

- `main` 是生产稳定分支。
- 日常功能开发使用 `feature/<topic>` 分支。
- 线上紧急修复使用 `hotfix/<topic>` 分支。
- 如果以后确实需要长期集成分支，再引入 `develop`。

## 日常工作流

1. 从干净的 `main` 开始。
2. 开始新任务前先同步 `origin/main`。
3. 每个任务单独建分支。
4. 变更集尽量小而清晰。
5. 在宣称完成前运行必要的验证。
6. 默认推送到 `origin`，不要推到 `upstream`。

## Git 安全规则

- 未经明确协调，不要 force-push 共享分支。
- 不要提交密钥、真实 token、生产密码等敏感信息。
- 不要改写已经发布的 `main` 历史。
- 不要把无关重构和功能修复混在同一个变更里。
- 需要同步上游时，必须有意识地处理 merge 或 rebase，并认真检查冲突。

## 文档规则

- 工作流有变动时，同时更新 `docs/engineering/`。
- 创建或调整执行计划时，同时更新 `docs/plans/`。
- 本文件必须保持简洁；一旦开始变成操作手册，就应迁移到 `docs/engineering/`。

## 对编码代理的要求

- 开始改代码前先读本文件。
- 优先遵守仓库已有约定，而不是套用通用默认值。
- 变更工作流、发布流程、部署方式时，要在同一任务中同步更新文档。
- 对远端或分支有疑问时，优先保持 `origin = fork`、`upstream = source` 这个模型不变。
